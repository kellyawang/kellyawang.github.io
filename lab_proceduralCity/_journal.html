Team Generation: Splines and Shots and Rendering for Large City Scenes

Matt and Devin - Spent most of the work day setting up splines and manually moving blocks within the full city for all the big city shots. The primary issue we encountered was the underlying plane. With our original skybox, the plane stood out too much against the horizon, though the other good skybox did not work well for many angles. Instead, we had to compromise and edit the skybox to look about halfway between the two. For interior shots, there is no issue. For wider shots, there is still some issue with there being too stark a dropoff, but for many shot it looks much, much better.

Then Matt, Devin, and Diwas had it set so that when we switched the scn file to be on Devin's faster computer, all the correct building OBJ's would be loaded in (since they coincidentally had the same filenames as the cubes on Matt's computer). So, we just had to load the scene to load the full, correct city. This slowed G3D to a crawl, and pressing F6 to start and stop was impossible. So, we set App.cpp to programmatically start and stop recording, and then we would have to click OK to save.

We rendered the first video, only to discover that there was an immense amount of strobing on the distant lights. We tried to fix this by adjusting the frame rate, but it had no effect. Prof. McGuire suggested that we supersample the scene, and then reduce the size after that. To do this, we increased the resolution size by 2 in each dimension, set to record halfsize, then ran again. The next issue we encountered, then, was that the G3D window was too big, and since we can only move it up so far (and the GUI was lagging too much to drag individual windows within G3D), we had no way of clicking OK after the video finished recording. Fortunately, Devin's retina macbook had an option to increase screen space, which then allowed enough room.

The program would crash after trying to render too many frames, and so we had to render a certain number of frames at a time, then save, then continue, in order to get the right shots.

Team Generation: Polygon Reduction

Diwas, Devin, and Matt attempted to reduce unused triangles by a number of means. First we tried removing duplicate vertices, because our grammars generated inefficient buildings by duplicating a lot of vertices and faces. We initially wrote a script, but it turns out that Blender had one built in. Unfortunately, this only reduced our total memory footprint by a couple percent, which didn’t help us out for loading up the buildings. Then we tried to find softwares that could compress the obj files for the generated buildings. We couldn’t find anything that was helpful. Our next guess was to try to remove hidden vertices by somehow isolating which ones they were. The first thought, though we realized it was unfeasible, was to raytrace such that we could detect those triangles which were never encountered, and remove them. Instead, we tried in Blender to choose invisible vertices. Blender could not detect them, but we realized there was another way to detect invisible vertices–turning the buildings over, and selecting visible vertices from within the building. This made a substantial impact on the size of the building, but it was unfeasible to perform this processing for each building, especially because we would have to do it very carefully so as not to accidentally select out-facing triangles. In the end, we were forced to simply rely on the optimizations we coded into building generation, via postprocessing or more careful generation.

Team Grammar: Buildings Complete

Kyle - Building Grammar - Got full grammar system done. The basic grammar element is an Expander that chooses from among a couple of different building types (as defined by rect placers with certain side and corner motifs), builds one of them, then returns another expander to the BuildingPlacer class to place again. BuildingPlacer determines how the Expanders are placed. Expanders define their building look and also what they might expand into next by returning a new type of Expander afterwards.

I then incorporated the different building types I was playing around with into this grammar system. I first ported the kit kat buildings - I had defined different rect placers and gave them to the constructor of the different bottom mid and top kk expanders. Afterwards, I requested several new window types from the building primitive team and made a new sets of expander types revolving around those window types.

For the below images, I spammed the generate next building button and took pictures of each result. The buildings are 20 x 30 x 20. They generate in about 10 seconds and take up around 6-9 mb.

Team Primitive: Textures Breakthrough!

After much work and stress, we finally figured out a way to texture.

After attempting to texture primitives using Blender, we decided to switch to Photoshop instead. We make a new 3D Layer from the .obj files, and color the Diffuse texture map as we like it. We can then export a 3D layer and Photoshop automatically updates the .obj, .mtl files, and some default jpegs for the Material and something called "Default IBL".

The color scheme we are using has the following coordinates:
Pale gray blue: 182, 194, 196
Light gray-blue: 121, 141, 145
Dark gray-blue: 54, 66, 69

We have textured the Cube, KitKat_sides, quarter_sphere, and Buttress, and the windows on the KitKat are emissive. We should also remember to make Cubes with colors of varying saturation to tile across building facades (if we have time).

Progress on Primitives:

We reached our goal of 25 primitives and now have created the building blocks for the building grammar group to use. We also now have a better understanding of complexity and details of complex materials. It is possible to create a simple building with a combination of multiple materials (each primitive is defined with its own .mtl file). We decided our scene would be the most effective and would fit our vision the most if we used simple dark concrete for building facades, and made windows and lamps emissive. It is possible to do this in Blender, the difficulty is rendering light properly, and adding post-processing effects (we should ask Morgan about this).

We found out how to make an emissive material! The attribute Ke in a .mtl file specifies the emissive color of the object. Blender does not include this variable in the .mtl file it saves out for some reason, which is why we were struggling with making windows emissive. (Sourceforge: http://cgkit.sourceforge.net/doc2/objmaterial.html, Morgan's blog: http://casual-effects.blogspot.com/2012/01/translucency-in-obj-mtl-files.html)

2014-10-10

Team City Layout: Grid Generation

Our grid generator works! Now we can generate buildings in a grid pattern.

Brief summary:

city.py will load a generator, which is currently gridGenerator.py, to generate a city. Each building is generated by calling the generateBuilding method in building.py, which takes 5 parameters, the xyz coordinates, the file name and a type class specified in buildingType.py.

To Grammar people: work on generateBuilding to provide grammar for different types on buildings, this is the method we will call. Make sure metaType=0 means the default cube building. Extend the type in anyway you prefer and document the extra parameters you need.

objectManipulator.py provides all basic functions you need to make an object. object.load() loads an obj file and store it. I current only implement the translation, and will add other transformations later, and will probably provide a helper method to do all the transformation at once. Use object.append() to append one object into another and use object.write() to write an obj file.

Workflow should be like this:

while more stuff in grammar

tmpobj.load(some primitive)


tmpobj.transform(Diwas, Kyle.... params..)


mainobj.append(tmpobj)
